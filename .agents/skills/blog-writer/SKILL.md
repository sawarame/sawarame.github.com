# ブログ執筆ガイドライン

## 1. 基本構成

1. **フロントマター (Frontmatter)**:
   - 必須項目: `slug`, `title`, `date`, `draft`, `tags`, `emoji`
   - `title`: 内容を的確に表すシンプルなものにする（例：入門記事に「使いこなす」などの大げさな表現は避ける）。
   - `emoji`: 記事のテーマや内容に合った適切な絵文字を必ず1つ選定する（Zennのように記事のアイキャッチとしてトップに大きく表示されるため重要）。
2. **導入 (Hook)**: 読者が抱える課題に共感し、この記事で解決できることを簡潔に述べる。
3. **本文 (Body)**:
   - 必要に応じて `:::tip` などの Docusaurus 特有のアドモニション（警告・注釈）を活用する。
   - 専門用語は、初出時に簡潔な説明を添える。
4. **結論 (Wrap-up)**: 学びを要約し、読者が次に取るべきアクション（ツールの試用など）を提示する。

## 2. 言い回し・トーン・内容の規約

- **見出しのスタイル**: 特段の理由がない限り、見出し（### 等）に「1. 」「2. 」といった**ナンバリングを振らない**こと。
- **AIの擬人化排除**: AIを「相棒」「パートナー」「話の通じる相手」のように擬人化したり、AIが「理解する」「考える」といった主観的な表現は避ける。代わりに「コンテキストを反映する」「回答精度を向上させる」「一貫性を保つ」といった、技術的に客観的な表現を用いる。
- **誇張・文学的表現の回避**: 結びの部分などで「たった一つのコマンドからすべてが始まります」「〜への扉を開く」といった、ドラマチックまたはカッコつけた文学的表現は使用しない。シンプルに実用的なネクストステップを提示するに留める。
- **NG表現**: 「〜してみましょう」「興味深かったのは」といった、AI生成テキストにありがちな余計な前振りや呼びかけを避ける。
- **推奨**: 専門家として客観的かつ簡潔な事実を述べる。です・ます調を基本とする。
- **コマンドの記述**: `/init` や `/memory` などのスラッシュコマンドを紹介する際は、通常のターミナル（sh/zsh）ではなく、**Gemini CLI の対話画面（チャットインターフェース）内**で実行するものであることが伝わるよう明記する。
- **自然な開発用語**: 直訳的な不自然な表現（例：「補助資産」）は避け、日本の開発現場で一般的に使われる自然な表現（例：「関連リソース」）を意識する。
- **汎用的な具体例**: 記事内で具体例（コードやユースケースなど）を提示する際は、プロジェクトに特化しすぎたニッチなものではなく、多くの読者が直感的に理解しやすい一般的な例（例：Reactコンポーネントの生成、テストの自動化など）を用いる。
- **プライバシー配慮**: 記事内で具体例を出す際は、個人情報や機密情報が含まれないよう徹底する。
- **不自然な直訳表現の回避**: 英語の直訳（例：「あなたの指示を待っています」「〜への扉を開く」など）や、AI特有の不自然な言い回しを避ける。日本の開発者が日常的に使用する、自然で客観的な日本語表現を心がける。
- **Markdown の太字表示**: 太字（`**`）を使用する際は、レンダリング失敗（太字にならずにアスタリスクが表示される等）を防ぐため、**太字タグの前後に必ず半角スペースを入れる**こと（例：`これは **強調** です`）。また、**太字を多用しない**こと。特に「主観的な強調」や「文章全体の強調」には使用せず、UI上のラベル、設定項目、ファイル名などの客観的な対象を指し示す場合にのみ限定して使用する。
- **内容の正確性とハルシネーション防止**: 技術的な内容は、誤解を招かないよう正確に記述する。**存在を確認していない公式ドキュメントURLや、存在しないコマンド（例：`agy doctor`など）を推測ででっち上げること（ハルシネーション）は絶対に避ける。** ツールやURLの存在は必ず事前に確認した上で記述すること。
- **専門用語と抽象表現の回避**: 「エージェント・ファースト」「エージェントハーネス」「デリゲート」といった抽象的なバズワードや専門用語は避け、「サブエージェントが裏側で並行作業する」といった、素人でも具体的な動作がイメージできる平易な言葉に翻訳して記述する。
- **共有・同期に関する表現の注意**: 「コンテキストが同期される」「記憶が引き継がれる」といった曖昧な表現は、「チャットの会話履歴まで同期される」という誤解を招きやすいため避ける。具体的に何が共有されるのか（例：「設定ファイルやルールが共有される」等）を明記し、引き継がれないもの（会話履歴など）は注記する。
- **リンクの活用**: 公式ドキュメント、信頼できる情報源、または参照している GitHub リポジトリ等へのリンクを積極的に提供し、読者の利便性を高める。
- **誤解を招く表現の回避**: 読者が誤解しやすい表現（例：「爆速」「日本一」「世界一」などの過度な期待を抱かせる表現）は避け、ツールの限界や注意点も明確に伝える。

## 3. MDX / Docusaurus 特有の記法

- **画像とキャプションの配置**: 画像を挿入し、その下に説明文（キャプション）を添える際は、隙間が空きすぎないように Docusaurus (Infima) のクラスとインラインスタイルを用いて以下のように記述する。
  ```markdown
  <div className="text--center" style={{ margin: '2rem 0' }}>

  ![画像の説明（alt）](./images/filename.webp)
  <small style={{ display: 'block', marginTop: '-1rem', color: 'var(--ifm-color-emphasis-600)' }}>キャプションテキスト</small>
  
  </div>
  ```

## 4. 執筆ワークフロー

1. 構成案（アウトライン）を作成し、人間に承認を得る。
2. 本文を下書き(`draft: true`)で執筆する。
3. 人間が内容を確認し、必要に応じて修正指示を出す。
4. 修正完了後全体の内容を再度確認し、矛盾点や不自然な表現がないかチェックする。必要に応じてさらに修正を行う。
5. 修正を反映し、公開準備が整ったら `draft: false` に変更する。
